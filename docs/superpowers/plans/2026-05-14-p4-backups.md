# P4 (Backups: pgBackRest client + store) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Give the cluster durable, scheduled backups. After P4, `make deploy` installs pgBackRest on the backup store host and on every postgres node, creates the stanza, wires PostgreSQL's `archive_command` to push WAL to the store over SSH, and installs systemd timers for weekly full + daily differential backups — all idempotent, all SELinux-enforcing.

**Architecture:** Two new thin roles. `roles/backup_client/` targets the `postgres` group: it installs pgBackRest, generates a dedicated per-cluster SSH keypair (the public half is collected by `backup_store`), renders `/etc/pgbackrest/pgbackrest.conf` pointing at the store, and sets PostgreSQL's `archive_command` via Patroni's dynamic config. `roles/backup_store/` targets the `backup_store` group (one host, by default colocated with `monitor`): it installs pgBackRest, owns the backup store at `/var/lib/pgbackrest`, authorizes the postgres nodes' SSH public keys, renders the server-side config, creates the stanza (`stanza-create` runs here because the backup store is local to this host — pgBackRest then SSHes into each postgres node to read `pg_control`), and installs systemd timers for scheduled backups. The two roles communicate through Ansible facts: `backup_client` runs first to generate keys, `backup_store` reads `hostvars[*].backup_client_ssh_pubkey` to populate `authorized_keys`. SSH auth is a generated keypair — no operator key management, no reuse of host keys.

**Tech Stack:** Ansible roles + `ansible.posix.firewalld` + `community.general.sefcontext` (both already used by P0/P1), pgBackRest from PGDG (`pgbackrest` RPM), `ansible.builtin.openssh_keypair` for the dedicated key, `community.postgresql.postgresql_query` to push `archive_command` through Patroni REST is **not** used — `archive_command` is set in `patroni.yml`'s `postgresql.parameters` (Patroni owns it) — see Task 11. Molecule + podman for the role tests, single-node and ha scenarios.

## File Structure

**New files (in `roles/backup_store/`):**

- `roles/backup_store/defaults/main.yml` — `backup_store_*` knobs: package name, store path (references generic `backup_store_path` from `group_vars/all.yml`), store user (`pgbackrest` for the pgBackRest backend), SSH settings, retention/schedule defaults, firewalld zone.
- `roles/backup_store/meta/main.yml` — galaxy_info, no role deps.
- `roles/backup_store/tasks/main.yml` — orchestrate: `_assert` → `_install` → `_ssh` → `_configure` → `_firewall` → `_stanza` → `_timers`.
- `roles/backup_store/tasks/_assert.yml` — assert `backup_store` group non-empty and size 1, store path mount present (warn not fail), postgres group non-empty.
- `roles/backup_store/tasks/_install.yml` — install `pgbackrest`, create the `pgbackrest` store user, create `/var/lib/pgbackrest` with SELinux fcontext + `restorecon`.
- `roles/backup_store/tasks/_ssh.yml` — create the store user's `.ssh`, populate `authorized_keys` from every postgres node's `backup_client_ssh_pubkey` fact.
- `roles/backup_store/tasks/_configure.yml` — render `/etc/pgbackrest/pgbackrest.conf` (server-side: one stanza, store path, retention, optional `secondary_store`).
- `roles/backup_store/tasks/_firewall.yml` — open SSH (port 22) from postgres nodes only, via firewalld rich rules.
- `roles/backup_store/tasks/_stanza.yml` — `pgbackrest stanza-create` + `check`, run on the `backup_store` host (the store is local here; pgBackRest reads the remote `pgN-host` entries over SSH itself). `failed_when` tolerant of "already exists".
- `roles/backup_store/tasks/_timers.yml` — install `pgbackrest-full` and `pgbackrest-diff` systemd service+timer units (and `pgbackrest-expire`, `pgbackrest-check`).
- `roles/backup_store/handlers/main.yml` — `Reload systemd for backup_store`.
- `roles/backup_store/templates/pgbackrest.conf.j2` — server-side config.
- `roles/backup_store/templates/pgbackrest-backup.service.j2` — parameterized oneshot unit (`%i` = `full`/`diff`).
- `roles/backup_store/templates/pgbackrest-backup.timer.j2` — parameterized timer.
- `roles/backup_store/templates/pgbackrest-maint.service.j2` — parameterized oneshot for `expire`/`check`.
- `roles/backup_store/templates/pgbackrest-maint.timer.j2` — parameterized timer.
- `roles/backup_store/README.md` — variables, what it owns, ordering.

**New files (in `roles/backup_client/`):**

- `roles/backup_client/defaults/main.yml` — `backup_client_*` knobs: package name, SSH key path/type, stanza name, backup store host resolution, archive settings.
- `roles/backup_client/meta/main.yml` — galaxy_info, no role deps.
- `roles/backup_client/tasks/main.yml` — orchestrate: `_assert` → `_install` → `_ssh` → `_configure` → `_archive`.
- `roles/backup_client/tasks/_assert.yml` — assert pgBackRest installable, Patroni running (per-host), `backup_store` group resolvable.
- `roles/backup_client/tasks/_install.yml` — install `pgbackrest`, create `/etc/pgbackrest` + log dir with SELinux fcontext.

# backup_store

The pgBackRest backup store host. Targets the `backup_store` group (one
host; by default colocated with `monitor`). Owns the backup store at
`/var/lib/pgbackrest`, accepts SSH public keys from the postgres nodes,
and runs scheduled backups via systemd timers that SSH into the current
Patroni leader.

## Inputs (from response file, via group_vars)

| Variable | Meaning | Default |
|---|---|---|
| `backup_store_path` | backup store directory | `/var/lib/pgbackrest` |
| `backup_stanza` | stanza name (one per cluster) | `{{ cluster_name }}` |
| `backup_store_user` | OS account owning the store | `pgbackrest` |
| `backup_retention_full` | full backups to keep | `4` |
| `backup_secondary_store` | optional S3 store config | unset |

## What this role owns

- The `pgbackrest` store user and its `~/.ssh/authorized_keys`.
- `/var/lib/pgbackrest` (directory, ownership, SELinux label).
- `/etc/pgbackrest/pgbackrest.conf` (server-side).
- The stanza (`pgbackrest stanza-create`, run on the backup store host — the
  store is local here, so this is where stanza-create belongs).
- `pgbackrest-full`, `pgbackrest-diff`, `pgbackrest-expire`,
  `pgbackrest-check` systemd service + timer units.

## What this role does NOT own

- PostgreSQL's `archive_command` — that's `backup_client` (set via Patroni).
- The dedicated SSH keypair — `backup_client` generates it; this role
  only authorizes the public half.
- Restore / PITR — see `playbooks/restore.yml` (separate plan).

## Ordering

`_assert` → `_install` → `_ssh` → `_configure` → `_firewall` →
`_stanza` → `_timers`. The `_ssh` step reads `backup_client_ssh_pubkey`
facts that `backup_client` set earlier in the same `make deploy` run.
`_stanza` runs after `_configure` so the server-side config (with the
`pgN-host` SSH entries) exists, and after `_ssh`/`_firewall` so the store
host can reach the postgres nodes.

## Idempotence

Second run is zero-change: package present, user present,
`authorized_keys` content-compared, config templated by content, timer
units templated by content.

## Tags

- `backup` — full role
- `backup,install` — package + store dir only
- `backup,config` — render config only
- `backup,firewall` — firewalld only
- `backup,service` — timer units only
````

## Task 5: backup_store install

- [ ] **Step 1: Write `_install.yml`**

```yaml
---
- name: Install pgBackRest and SELinux helpers
  ansible.builtin.dnf:
    name: "{{ [backup_store_package] + backup_store_support_packages }}"
    state: present

- name: Create the pgBackRest backup store group
  ansible.builtin.group:
    name: "{{ backup_store_group }}"
    system: true
    state: present

- name: Create the pgBackRest backup store user
  ansible.builtin.user:
    name: "{{ backup_store_user }}"
    group: "{{ backup_store_group }}"
    home: "{{ backup_store_user_home }}"
    shell: /bin/bash
    system: true
    create_home: true
    state: present

- name: Ensure config directory exists
  ansible.builtin.file:
    path: "{{ backup_store_config_dir }}"
    state: directory
    owner: "{{ backup_store_user }}"
    group: "{{ backup_store_group }}"
    mode: "0750"

- name: Ensure the backup store directory exists
  ansible.builtin.file:
    path: "{{ backup_store_path }}"
    state: directory
    owner: "{{ backup_store_user }}"
    group: "{{ backup_store_group }}"
    mode: "0750"

- name: Ensure the log directory exists
  ansible.builtin.file:
    path: "{{ backup_store_log_path }}"
    state: directory
    owner: "{{ backup_store_user }}"
    group: "{{ backup_store_group }}"
    mode: "0750"

- name: Register SELinux fcontext for the backup store directory
  community.general.sefcontext:
    target: "{{ backup_store_path }}(/.*)?"
    setype: var_t
    state: present
  register: backup_store_sefcontext
  when: ansible_facts.selinux.status == "enabled"

- name: Relabel the backup store directory if fcontext changed
  ansible.builtin.command:
    cmd: "restorecon -RF {{ backup_store_path }}"
  when:
    - ansible_facts.selinux.status == "enabled"
    - backup_store_sefcontext.changed
  changed_when: true
```

current Patroni leader only, it sets PostgreSQL's `archive_command`
through Patroni's dynamic config and creates the stanza.

## Inputs (from response file, via group_vars)

| Variable | Meaning | Default |
|---|---|---|
| `backup_stanza` | stanza name | `{{ cluster_name }}` |
| `backup_store_user` | store user on the store host | `pgbackrest` |
| `postgres_osdba` | OS account running PostgreSQL | `postgres` |

## What this role owns

- The `pgbackrest` RPM on each postgres node.
- `/var/lib/pgsql/.ssh/id_pgbackrest{,.pub}` (the dedicated keypair).
- `/etc/pgbackrest/pgbackrest.conf` (client-side).
- PostgreSQL `archive_mode` / `archive_command` (via Patroni dynamic config).

## What this role does NOT own

- The backup store directory or `authorized_keys` — that's `backup_store`.
- Stanza creation — that's `backup_store` (the store is remote from the
  client's perspective, so `stanza-create` runs on the backup store host).
- Scheduled backup timers — that's `backup_store`.
- Restore / PITR — separate plan.

## Ordering

`_assert` → `_install` → `_ssh` → `_configure` → `_archive`. Runs
**before** `backup_store` in `site.yml` so the
`backup_client_ssh_pubkey` fact exists when the store builds
`authorized_keys`.

## Idempotence

Second run is zero-change: package present, keypair present (not
regenerated), config content-compared, `archive_command` already set in
Patroni config, stanza-create tolerates "already exists".

## Tags

- `backup` — full role
- `backup,install` — package only
- `backup,config` — config + SSH key only
- `backup,service` — archive_command + stanza only
````

    retention = backup.get("retention", {})
    if retention:

        full = retention.get("full")
        if full is not None and (not isinstance(full, int) or full < 1):

# Day-2 backups

Backups are declarative through the response file's generic `backup:`
block. P4 supports `tool: pgbackrest`; future plans may add additional
tools under the same contract. Edit the response file, then
`make deploy`. To target only the backup step, pass `--tags backup`.

## Trigger a manual backup

Backups run on the backup store host, which SSHes into the current
Patroni leader. To run one by hand:

```bash
# On the backup store host, as the pgbackrest user:
sudo -iu pgbackrest pgbackrest --stanza=<cluster_name> --type=full backup
sudo -iu pgbackrest pgbackrest --stanza=<cluster_name> --type=diff backup
```

Or trigger the systemd unit directly:

```bash
sudo systemctl start pgbackrest-backup@full.service
```

## Read backup status

```bash
sudo -iu pgbackrest pgbackrest --stanza=<cluster_name> info
```

The `archive` section shows the WAL range; `backup` lists each full and
differential with timestamps and sizes.

## Change retention

```yaml
backup:
  enabled: true
  tool: pgbackrest
  retention:
    full: 8        # keep 8 full backups instead of 4
```

`make deploy` re-renders the server-side config. The next `expire` timer
run (or a manual `pgbackrest expire`) enforces the new policy.

## Change the schedule

```yaml
backup:
  tool: pgbackrest
  schedule:
    full: "0 2 * * 0"          # Sundays at 02:00
    differential: "0 2 * * 1-6"
```

The role maps these onto systemd `OnCalendar` expressions on the
backup store host.

## Enable an S3-compatible second store

```yaml
backup:
  enabled: true
  tool: pgbackrest
  secondary_store:
    enabled: true
    type: s3
    bucket: my-pg-backups
    endpoint: s3.us-east-1.amazonaws.com
    region: us-east-1
    access_key: !vault | ...
    secret_key: !vault | ...
```

`make deploy` writes the secondary_store stanza into the server-side config and a
`0600` credentials file owned by the `pgbackrest` user. Backups then
push to both stores.

## Common gotchas

- **`archive_mode` pending restart**: the first time archiving is
  enabled, Patroni needs a restart. Run
  `patronictl restart <cluster_name> --pending` in a maintenance window.
- **SSH push fails**: the `backup_client` role generated a dedicated
  keypair at `/var/lib/pgsql/.ssh/id_pgbackrest`; the `backup_store`
  role authorized it. If you rebuilt a postgres node, re-run
  `--tags backup` so the new key is re-authorized.
- **Store disk full**: `/var/lib/pgbackrest` must be sized for
  `retention.full` full backups plus the WAL between them. Preflight
  warns if it is not a separate mount.
- **Restore / PITR**: not covered here — handled by
  `playbooks/restore.yml` (separate runbook).
```

3. **Variable / type consistency.** Operator-facing names are generic: the response file uses `backup:` and the generated vars are `backup_enabled`, `backup_tool`, `backup_stanza`, `backup_store_user`, `backup_ssh_port`, `backup_schedule_*`, `backup_schedule_*_oncalendar`, `backup_retention_full`, and `backup_secondary_store`. The pgBackRest backend consumes those through role-prefixed aliases. `backup_store_stanza` / `backup_client_stanza` both resolve from `backup_stanza` — same value, role-prefixed alias, consistent. `backup_client_ssh_pubkey` is set as a fact in `backup_client/_ssh.yml` (Task 9) and read in `backup_store/_ssh.yml` (Task 13) and `backup_store/templates/pgbackrest.conf.j2` (Task 14) — name matches everywhere. `backup_client_pg_user` / `backup_client_pg_path` / `backup_client_pg_port` are set in `backup_client` defaults (Task 6) and read cross-host in the store's config template via `hostvars[host].backup_client_pg_*` (Task 14) — consistent. `backup_store_user_home` defined Task 2, used Tasks 5, 13. `timer_instance` is passed via `vars:` in Task 16's `_timers.yml` loop and consumed in the `.timer.j2` templates — consistent. The systemd instance names (`full`, `diff`, `expire`, `check`) are used consistently across templates, `_timers.yml`, and the `ha` verify.

4. **Where stanza-create runs, and why client-then-store.** The backup store is remote from every postgres node's perspective (`repo1-host` points at the `backup_store` host). pgBackRest's `stanza-create` must run where the backup store is *local* — on the `backup_store` host — and from there pgBackRest SSHes into each postgres node (via the `pgN-host` entries in the server-side config) to read `pg_control`. So stanza creation is a `backup_store` task (Task 12 → `roles/backup_store/tasks/_stanza.yml`), ordered after `_configure.yml` (server-side config with the `pgN-host` entries must exist) and after `_ssh.yml`/`_firewall.yml` (the backup store host must be able to reach the nodes). The `backup_client` role therefore does **not** create the stanza; its orchestration (Task 8) ends at `_archive.yml`, and it keeps an empty `handlers/main.yml` only for layout parity. This also settles the playbook order: `site.yml` imports `_backup_client.yml` before `_backup_store.yml` because (a) the client must generate its keypair and export `backup_client_ssh_pubkey` before the store can build `authorized_keys`, and (b) nothing in the client role needs the store to have run first — the client's only backup-store-touching concern, stanza-create, was moved to the store side. Import-playbook facts persist across plays in one run, so the fact hand-off works in a single `make deploy`.

5. **SELinux.** `backup_store` store dir gets `var_t` (generic; pgBackRest has no dedicated policy type on EL10 — verify with `semanage fcontext -l | grep pgbackrest` during execution and prefer a dedicated type if one exists). `backup_client` log dir gets `postgresql_log_t` since pgBackRest client logs sit alongside PG logs. Both follow the P1 etcd pattern: `sefcontext` + conditional `restorecon`. If a runtime AVC denial appears in molecule, the scenario will fail loudly (spec §8.7) — do not add a blanket permissive domain.

6. **What's deliberately out of scope.** `playbooks/restore.yml` / PITR (separate plan), S3 `secondary_store` end-to-end CI fixture (config-render tested only), `pgbackrest_exporter` (P5), backup encryption, multi-host store HA. The `_assert.yml` in Task 4 hard-fails on `backup_store` group size != 1 to make the last one explicit.
