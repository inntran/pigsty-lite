# P1 (etcd) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stand up an etcd Raft cluster that is healthy, mTLS-secured by the pigsty-lite internal CA, and ready to serve as Patroni's DCS in P2. One inventory group, two profile shapes: 1-member etcd for `single`, 3-member colocated quorum for `ha`.

**Architecture:** A new thin role `roles/etcd/` installs the `etcd` RPM from PGDG, lays down `/etc/etcd/etcd.conf.yml` from a template, registers an SELinux fcontext for the data dir (vendor default `/var/lib/etcd`), opens firewalld services `etcd-client` and `etcd-server` restricted to the `etcd` and `postgres` inventory groups, and bootstraps the cluster with `initial-cluster-state: new` (`existing` for member-add later). Peer + client traffic both use the per-host cert already issued by the P0 `certs` role; CN = `inventory_hostname` and SANs already include host DNS + IP. A new `_etcd.yml` playbook orchestrates install → start → health gate. Idempotence is guaranteed by Molecule's built-in step.

**Tech Stack:** Ansible role + Jinja2 templates, etcd RPM from PGDG, systemd, firewalld rich rules, SELinux fcontext via `community.general.sefcontext`, `community.crypto` (already in P0), Molecule + podman for tests.

## File Structure

**New files (in `roles/etcd/`):**

- `roles/etcd/defaults/main.yml` — port, data dir, TLS file paths, peer/client URL templates, RPM package name, image of inventory group → cluster initial-cluster string.
- `roles/etcd/meta/main.yml` — role metadata, dependencies (none — `repos` and `certs` are run earlier in `site.yml`, not as role deps).
- `roles/etcd/tasks/main.yml` — orchestrates: assert preconditions → install RPM → fcontext + mkdir → render config → render systemd drop-in (if needed) → enable+start unit → wait for health.
- `roles/etcd/tasks/_assert.yml` — fail fast if etcd group size is 0 or doesn't match profile; verify cert files exist from P0.
- `roles/etcd/tasks/_install.yml` — dnf install + selinux fcontext + dirs.
- `roles/etcd/tasks/_configure.yml` — render `/etc/etcd/etcd.conf.yml`, systemd `EnvironmentFile` if used.
- `roles/etcd/tasks/_service.yml` — enable+start, wait for `etcdctl endpoint health`.
- `roles/etcd/tasks/_firewall.yml` — open `etcd-client` and `etcd-server` services with source CIDRs derived from `etcd` + `postgres` host IPs.
- `roles/etcd/templates/etcd.conf.yml.j2` — single source of truth for etcd config; renders correctly for both 1-member and N-member clusters.
- `roles/etcd/README.md` — what the role does, variables, gotchas.

**New files (playbooks / inventory wiring):**

- `playbooks/_etcd.yml` — new internal playbook that imports the role against `etcd` group.

- [ ] **Step 2: Write README.md**

```markdown
# etcd

Install etcd from PGDG, render `/etc/etcd/etcd.conf.yml`, open firewalld
services `etcd-client` and `etcd-server` to peers + clients, and gate on
`etcdctl endpoint health` before returning.

## Profile shapes

- `single`: 1 member, no quorum, `initial-cluster-state: new`. Adequate
  for `single` deployments where the host is the failure domain anyway.
- `ha`: 3 members colocated on the postgres hosts, on a different block
  device than `/var/lib/pgsql/*/data` per the design.

## TLS

Both peer (2380) and client (2379) endpoints require TLS and client
certificate authentication. The role reuses the per-host cert issued by
the P0 `certs` role; CN = `inventory_hostname`. Peer-to-peer auth works
because each peer presents its own host cert and trusts the same CA.

## Variables

See `defaults/main.yml`. Override paths only with strong reason; SELinux
fcontext rules are tied to the vendor default `/var/lib/etcd`.

## Firewalld

Opens built-in services `etcd-client` and `etcd-server` with rich rules
restricting source addresses to the `etcd` group (peer) and `etcd` plus
`postgres` group (client). Nothing else.
```

- [ ] **Step 3: `roles/etcd/tasks/_install.yml`**

```yaml
---
- name: Install etcd package
  ansible.builtin.dnf:
    name: "{{ etcd_package }}"
    state: present

- name: Ensure config directory exists
  ansible.builtin.file:
    path: "{{ etcd_config_dir }}"
    state: directory
    owner: root
    group: etcd
    mode: "0750"

- name: Ensure data directory exists with correct ownership
  ansible.builtin.file:
    path: "{{ etcd_data_dir }}"
    state: directory
    owner: etcd
    group: etcd
    mode: "0700"

- name: Register SELinux fcontext for etcd data dir
  community.general.sefcontext:
    target: "{{ etcd_data_dir }}(/.*)?"
    setype: etcd_var_lib_t
    state: present
  register: etcd_sefcontext

- name: Relabel etcd data dir if fcontext changed
  ansible.builtin.command:
    cmd: "restorecon -RF {{ etcd_data_dir }}"
  when: etcd_sefcontext.changed
  changed_when: true
```

- [ ] **Step 2: If failure, debug and fix in role files (not in the scenario).**

Likely failure modes and fixes:

- `etcd` package not found → confirm PGDG repo RPM is installed by `prepare.yml`. The PGDG repo carries etcd for RHEL family.
- SELinux denial on data dir → confirm `community.general.sefcontext` was actually applied and `restorecon` ran. Check `ausearch -m AVC -ts recent`.
- Health gate times out → check `journalctl -u etcd`; common cause is the advertise URL not being reachable from the listening interface (in podman, the container IP may differ from `ansible_host`). The defaults in `_install.yml` and `etcd.conf.yml.j2` use `ansible_host` for advertise and `network_any_address` for listen, which works in podman because `ansible_host=127.0.0.1`. If it fails here, this is a scenario problem, not a role problem.

## Self-review notes

1. **Spec coverage check.** The design spec calls for: etcd RPM install, `/etc/etcd/etcd.conf.yml`, systemd unit, TLS, post-install `etcdctl endpoint health` gate, kubespray-modeled role, 1 or 3 members tied to profile, separate block device for `/var/lib/etcd`, firewalld `etcd-server` + `etcd-client`, SELinux fcontext for `/var/lib/etcd`. Each is addressed by Task 1 (defaults), Task 5 (`_install.yml` + `_configure.yml` + `_service.yml`), Task 5 (`_firewall.yml`), Task 7 (template), Task 8/10 (health gate verification), Task 4/9 (Molecule single and ha scenarios). Block-device separation for `/var/lib/etcd` is the operator's responsibility per the design (README is explicit). P1 does not add a preflight check for this — the failure mode (same disk as PG data) is a performance issue, not a correctness issue, and adding a "is this on a separate block device" check across cloud providers is more code than it's worth. The smoke-test step (Task 15) reminds the operator to verify.

2. **Placeholder scan.** No `TBD`, `TODO`, `implement later`, or "Similar to Task N" references. Every code block is concrete.

3. **Variable / type consistency.** `etcd_cert_file`, `etcd_key_file`, `etcd_trusted_ca_file`, `etcd_data_dir`, `etcd_advertise_address`, `etcd_listen_address`, `etcd_initial_cluster_state`, `etcd_member_name` are defined in Task 1 (defaults) and used identically in Task 5 (tasks), Task 7 (template), and verify steps. Port names match `all.yml`: `etcd_client_port`, `etcd_peer_port`. `etcd_initial_cluster` is removed from `group_vars/etcd.yml` in Task 10 in favor of inline template rendering — Task 7's template includes the variable form initially; Task 10 corrects it. (The two-step approach lets the implementer learn the failure mode rather than memorize a fix.)

4. **kubespray reference.** The spec says "modeled on kubespray's etcd role." That's a stylistic reference, not a code copy. The role here is much smaller (Patroni DCS only, no etcd-events, no client lib install, no operator ergonomics). Codex should *read* kubespray's etcd role if it wants for context, not vendor it.

5. **What's out of scope for P1 (do not implement):**
   - Automated etcd member add/remove (will be a runbook in P6).
   - etcd backup (covered by Patroni's own DCS migration story, not etcd's own backup; PG backups go through pgBackRest in P4).
   - etcd metrics scraping by VictoriaMetrics (P5).
   - Patroni install or config (P2).
