# CA Operating Manual

This document covers the operation, maintenance, and troubleshooting of the pigsty-lite Certificate Authority (CA).

## Overview

Pigsty-lite uses a self-signed CA to issue certificates for internal services (etcd, Patroni, Postgres, HAProxy, nginx, Prometheus, etc.). The CA is generated once on the control node and its certificates are distributed to all cluster nodes.

- **CA Location (Control Node):** `pki/ca/ca.crt` and `pki/ca/ca.key`
- **CA Distribution (All Nodes):** `/etc/pki/pigsty/ca.crt` and `/etc/pki/pigsty/ca.key`
- **Per-Host Certificates:** `/etc/pki/pigsty/{{ hostname }}.crt` and `/etc/pki/pigsty/{{ hostname }}.key`

---

## Part I: Initial CA Setup

### 1.1 Prerequisites

- Ansible with `community.crypto` collection installed
- `openssl` command-line tool available on the control node
- Write access to `pki/` directory on the control node

### 1.2 Generate the CA

The CA is generated during initial cluster deployment by running the `_ca.yml` playbook:

```bash
cd /path/to/pigsty-lite
ansible-playbook playbooks/_ca.yml -i inventory/
```

**What happens:**

1. The control node generates a 4096-bit RSA private key (`pki/ca/ca.key`)
2. A self-signed certificate is created with a 10-year validity (`pki/ca/ca.crt`)
3. The CA certificate is distributed to all nodes in `/etc/pki/pigsty/ca.crt`
4. The CA key is also copied to `/etc/pki/pigsty/ca.key` for renewal purposes

**Output:**

```
pki/ca/
├── ca.crt     (distributed to all nodes)
├── ca.key     (distributed to all nodes)
└── .gitkeep
```

### 1.3 Verify CA Generation

After generation, verify the CA certificate:

```bash
# View certificate details
openssl x509 -in pki/ca/ca.crt -text -noout

# Check validity dates
openssl x509 -in pki/ca/ca.crt -noout -dates
```

Expected output:

```
notBefore=...  (generated today)
notAfter=...   (10 years in the future)
```

## Part II: Certificate Management

### 2.1 Generate Per-Host Certificates

Per-host certificates are generated by the `certs` role. This happens automatically during `ansible-playbook playbooks/_provision.yml`.

**Process:**

1. A private key is generated on each target node: `/etc/pki/pigsty/{{ hostname }}.key`
2. A Certificate Signing Request (CSR) is generated on the target
3. The CSR is transferred back to the control node
4. The CA signs the CSR on the control node
5. The signed certificate is distributed back to the target: `/etc/pki/pigsty/{{ hostname }}.crt`

**Manual invocation** (if needed):

```bash
ansible-playbook playbooks/certs.yml -i inventory/ --tags certs
```

### 2.2 Certificate Validity and Renewal

- **Default validity:** 730 days (2 years)
- **Renewal window:** 30 days (certs are renewed if `notAfter < now + 30 days`)
- **Manual renewal:** Re-run the certs playbook to trigger renewal logic

Check a certificate's expiration:

```bash
# On any node
openssl x509 -in /etc/pki/pigsty/{{ hostname }}.crt -noout -dates
```

## Part III: Troubleshooting

### 3.1 Certificate Not Found

**Error:** `certificate not found at /etc/pki/pigsty/hostname.crt`

**Diagnosis:**

1. Check if `/etc/pki/pigsty/` exists and is readable:

   ```bash
   ssh user@node ls -la /etc/pki/pigsty/
   ```

2. Check if the hostname matches (case-sensitive):

   ```bash
   ssh user@node hostname
   ```

**Solution:**

- Re-run the `certs` role to generate the missing certificate
- Ensure `inventory_hostname` matches the actual hostname

### 3.2 Certificate Validation Fails

**Error:** `certificate verification failed`

**Diagnosis:**

1. Verify the CA certificate is present and correct on all nodes:

   ```bash
   ssh user@node cat /etc/pki/pigsty/ca.crt | openssl x509 -noout -text
   ```

2. Verify the per-host certificate is signed by the CA:

   ```bash
   openssl verify -CAfile pki/ca/ca.crt pki/certs/hostname.crt
   ```

**Solution:**

- Ensure `/etc/pki/pigsty/ca.crt` is world-readable (`644`)
- Re-run the `_ca.yml` playbook to re-distribute the CA certificate
- Check system time synchronization (certificates are time-sensitive)

### 3.3 Expired Certificate

**Error:** `certificate has expired`

**Diagnosis:**

```bash
openssl x509 -in /etc/pki/pigsty/hostname.crt -noout -dates
```

**Solution:**

1. Trigger certificate renewal:

   ```bash
   ansible-playbook playbooks/certs.yml -i inventory/ --tags certs
   ```

2. Restart affected services (they will pick up the new certificate):

   ```bash
   # For etcd
   systemctl restart etcd
   
   # For Patroni
   systemctl restart patroni
   ```

### 3.4 Insufficient Permissions

**Error:** `permission denied` when reading private keys

**Diagnosis:**

```bash
ssh user@node ls -la /etc/pki/pigsty/*.key
```

**Solution:**

- Ensure keys are readable by the service user (usually `600` or `640` with group ownership)
- For services that need to read keys, add them to the appropriate group:

  ```bash
  ssh user@node usermod -aG pigsty etcd
  ```

### 3.5 CA Key Compromise

**Severity:** Critical

If the CA private key (`/etc/pki/pigsty/ca.key` or `pki/ca/ca.key`) is compromised:

1. **Immediate action:** Revoke all certificates
2. **Generate new CA:**

   ```bash
   rm -rf pki/ca/
   ansible-playbook playbooks/_ca.yml -i inventory/
   ```

3. **Regenerate all per-host certificates:**

   ```bash
   # Remove old certificates (optional; new ones will overwrite)
   ansible-playbook playbooks/certs.yml -i inventory/ --tags certs
   ```

4. **Restart all services** to load new certificates
5. **Audit** which deployments used the compromised CA

## Part IV: Backup and Recovery

### 4.1 Backup CA Keys

**Location to protect:** `pki/ca/ca.key` and `/etc/pki/pigsty/ca.key`

Recommended backup procedure:

```bash
# Create encrypted backup
tar -czf ca-backup-$(date +%Y%m%d).tar.gz pki/ca/
gpg -c ca-backup-*.tar.gz  # Encrypt with a passphrase

# Store in secure location (off-site, offline storage)
```

### 4.2 Restore CA Keys

If the control node is lost but CA backups exist:

```bash
# Restore from backup
gpg -d ca-backup-*.tar.gz.gpg | tar -xz

# Verify CA integrity
openssl x509 -in pki/ca/ca.crt -noout -text
```

### 4.3 Audit Trail

- **CA generation date:** Check `pki/ca/ca.crt` metadata
- **Certificate issues:** Stored CSRs (if preserved) in `pki/certs/`
- **Ansible logs:** Review `ansible.log` for certificate generation events

---

## Part V: Security Best Practices

### 5.1 CA Key Protection

- **Never share** `pki/ca/ca.key` outside the control node and necessary renewal processes
- **Rotate access:** Limit who can read `pki/ca/` to ansible users only
- **Permissions:** Keep `pki/ca/ca.key` at `0600` (readable by owner only)

### 5.2 Certificate Distribution

- **Automate:** Always use Ansible to distribute certificates; never manually copy
- **Audit:** Track which nodes have which certificates via Ansible inventory
- **Validation:** Use `openssl verify` to confirm all certs are signed by the CA

### 5.3 Monitoring

Set up alerts for:

- Certificates expiring in < 30 days (enable monitoring role)
- Certificate validation failures in service logs
- Unauthorized access attempts to `/etc/pki/pigsty/ca.key`

## Part VI: Reference Commands

### Check CA Certificate

```bash
openssl x509 -in pki/ca/ca.crt -text -noout
```

### Check Per-Host Certificate

```bash
openssl x509 -in /etc/pki/pigsty/hostname.crt -text -noout
```

### Verify Chain

```bash
openssl verify -CAfile pki/ca/ca.crt /etc/pki/pigsty/hostname.crt
```

### Check Expiration

```bash
openssl x509 -in /etc/pki/pigsty/hostname.crt -noout -dates
```

### List All Certificates

```bash
# On any node
ls -la /etc/pki/pigsty/*.crt
```

### Validate All Per-Host Certs on Control Node

```bash
for cert in pki/certs/*.crt; do
  echo "Verifying $cert..."
  openssl verify -CAfile pki/ca/ca.crt "$cert"
done
```
